* Cooperation

* Historical background
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)

    - Unsupported printer driver...
    - Identified the root cause:
      - Knowledge is power
  - How to take the power back ?
    - Share knowledge, share the code
    - Code availability is not enough !
    - A "social" contract between programmers
  - GPL license, GNU Project started, FSF

* Up/Down/stream loop
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Share patches upstream first
    - hold downstream merge
    - Listen to Early Upstream feedback
    - Is change is desirable? the right way?
    - Improve, Adapt, Split, Refactor
  - Downstream merge
  - Rebase loop (may be a long race)
    - Technical dept will grow over time
    - Reduce gap until fully aligned

* Context is gold
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Never break the evolution chain
  - Preserve history, authorship
    - no cut and paste
  - Good (external) dependencies management
    - watch upstream versions
  - Tracking patches
    - git metadata (hint: debian DEP3)

* Metadata
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)

  - Helpful commit messages:
    #+BEGIN_SRC

git commit -sam ‘context: Add X for feature Y...
Because of Z reason...
Bug: url://upstream/project/bug/42
'
    #+END_SRC

  - https://www.conventionalcommits.org/en/v1.0.0/

* Linking contexts
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Trackers might be updated after commits:
    - regression, metrics, cross links...
  - Use git commit footer for metadata:
    - Bug: $url (Upstream context)
    - Bug-$downstream: $url or $id (downstream context)
    - Origin: $url (Where patch was published first)
    - Forwarded: $url (Where upstream will review it)
  - https://dep-team.pages.debian.net/deps/dep3/

* Attribution
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Respect authors (and their works or time), in commit messages:
    - Author: ...
    - Thanks-to:, Credit-to:, Reported-by:, Suggested-by: ...
  - Author is the most knowledgeable
    - and may help later
  - Signed commits may help to insure integrity
    - Legal reasons, Copyright holders

* Legal
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Never assume that random public code is safe
    - Minimal chain of trust to author should be established
  - FLOSS is not public domain: Rights and duties
    - Different philosophies, and roles
    - Author/User, Business/Community, OSI/FSF, Permissive/Copyleft...
  - https://reuse.software/
    - tools for SBOM, SPDX, Copyrights etc

* Security matters
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Public code have to be scanned for vulnerability or legal compliance
    - vulnerabilities reported (1st private, then public) and handled
  - Downstream code may be less exposed
    -  Patches may fix ? or add more vulnerabilities
  - Scanning code, verifying code is long and costly
    - Share effort in the upstream

* OSS organizations
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Vendor neutral, Hold copyrights
  - Funded by companies / individuals
  - Provide infrastructures
  - Training and certifications
  - Many: LF, Apache, Eclipse, FSF, Mozila...

* Abandonware
  #+ATTR_REVEAL: :frag (fade-in fade-in fade-in fade-in)
  - Projects may became inactives
    - lack of funding or interest
  - Seriaous Problem will appear
  - Prefer to shift to best effort co-maintenance ASAP
  - https://abandonware.github.io
    - Community maintained packages
    - Maximize benefits, minize effort
    - No trade of on security

* Summary
  #+ATTR_REVEAL: :frag (fade-in)
  - Avoid "Not Invented Here"
    - easy to start / harder to maintain
  - Join relevent open projects
    - Learn by doing
    - Help community
  - Minimize downstream changes
  - Establish long term strategies with communities
    - Prefer Invest in co-creation of value than troubleshooting
  - Cooperate to minimize effort, maximize results
